## Background

In large-scale automotive software projects, automated end-to-end testing is critical for both regression and feature validation. Robot Framework is a popular choice for keyword-driven testing, but as test coverage grows, so does the volume and complexity of logs generated by both the system under test and the test framework itself.

*Problem:* How do we automatically detect, classify, and report relevant log events (errors, warnings, anomalies) from massive log files, without manual review?

---

## Why Log Detection Matters

- *Early bug detection:* Many failures are only visible in logs, not as test case failures
- *Faster triage:* Automated log parsing highlights root causes for failed or flaky tests
- *Continuous Integration:* Enables real-time feedback in CI pipelines, reducing time-to-fix

---

## Architecture Overview

+-------------------+         +-------------------+         +-------------------+
|  Robot Framework  |  --->   |  Log Detector     |  --->   |  Report/Alert     |
|  Test Execution   |         |  (Python script)  |         |  (HTML, Email, CI)|
+-------------------+         +-------------------+         +-------------------+

- *Robot Framework* runs test suites, generates output logs (XML, HTML, plain text)
- *Log Detector* is a Python tool that parses logs, applies pattern matching, and extracts events
- *Report/Alert* summarizes findings for developers/testers

---

## Implementation Patterns

### 1. Pattern-based Detection
- Use regular expressions to match error, warning, or custom patterns in log files
- Example: r"(ERROR|FAIL|EXCEPTION|CRITICAL)"

### 2. Keyword-driven Filtering
- Allow users to specify custom keywords or ignore lists per project
- Example: Ignore known benign warnings, focus on new/unexpected ones

### 3. Context Extraction
- For each match, extract surrounding lines (stack trace, timestamp, test case)
- Helps with root cause analysis

### 4. Integration with Robot Framework Output
- Parse output.xml for structured test results
- Correlate log events with specific test cases or keywords

### 5. CI/CD Integration
- Run log detection as a post-processing step in Jenkins/GitLab CI
- Fail the build or raise alerts if critical issues are found

## Example: Practical Log Detector (small, slightly messy but real)

This is a small script I used during debugging. It's not perfect — a bit quick-and-dirty — but it worked and shows real steps.

import re
import json

def detect_log_events(logfile, patterns):
    with open(logfile) as f:
        lines = f.readlines()

    events = []
    for i, line in enumerate(lines):
        for pat in patterns:
            if re.search(pat, line):
                context = lines[max(0, i-2):i+3]
                # simple normalize: strip ms suffix if present
                norm = line.strip().replace('[ms]', '')
                events.append({
                    'match': norm,
                    'context': ''.join(context)
                })

    # write a small summary file (quick hack)
    with open('log_events_summary.json', 'w') as out:
        json.dump({'count': len(events), 'events': events[:20]}, out, indent=2)
    print(f'Found {len(events)} events, wrote log_events_summary.json')

if __name__ == '__main__':
    patterns = [r"ERROR", r"FAIL", r"EXCEPTION"]
    detect_log_events('logs/output_2026-03-25.log', patterns)

Run this locally on the CI artifact and check log_events_summary.json. I ran this on an ubuntu-20.04 runner; the whole parse took ~2m on our large logs. It's a bit messy, yes — but practical.

## Author's Anecdote & Snapshot (anonymized)

While integrating the detector into CI, a flaky test repeatedly passed locally but failed on CI with no obvious error. The log detector flagged a recurring warning during the teardown that correlated with a race condition in a mock service. After anonymizing and collecting the relevant log window, a focused fix reduced the flakiness dramatically.

The CI run was on Jenkins using the job ci/run-tests. I reproduced the failure with robot --output output.xml tests/smoke.robot and ran it five times to confirm the pattern. Why did it only fail on CI? The environment had a timing difference in a teardown mock. It was annoying. After the fix the tests became stable.
